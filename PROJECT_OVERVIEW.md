# 城味项目总览

城味是一个围绕“本地生活消费 + 店铺运营 + 内容分享”构建的完整前后端项目。它不是单纯的探店展示站，也不是只做下单的团购系统，而是把用户发现店铺、购买优惠券、生成订单、到店核销，以及管理员建店、店长管店、店员核销这些链路串成了一套可以闭环运行的业务系统。项目的核心目标有两个：第一，让普通用户能够像使用一个轻量本地生活平台一样浏览店铺、抢券下单、写探店笔记；第二，让门店和平台侧拥有各自清晰的后台能力，形成真正可管理、可维护、可扩展的三端体系。

从角色视角看，这个项目一共覆盖了四类身份：普通用户、店员、店长、平台管理员。普通用户负责浏览、搜索、下单、支付、核销前查看自己的订单与核销码，同时还能发布博客、评论店铺、关注他人、查看关注流；店员负责订单核销与本店订单查询；店长在店员能力之上，还拥有维护本店资料、创建本店员工的权限；平台管理员则负责建立新店铺并指定首个店长，从平台层面初始化整个门店组织结构。这个分层的价值在于，权限不是简单靠前端隐藏按钮完成的，而是从接口入口、登录态隔离、服务层校验一路收口到后端逻辑里，使每个角色只做自己该做的事情。

整个系统的后端是一个典型的 Spring Boot 单体应用，但它不是简单的 CRUD 项目。项目主体采用 Spring Boot + MyBatis-Plus + MySQL + Redis 的组合来实现数据持久化、缓存、登录态、秒杀、GEO 附近查询和 Feed 流。Web 层由多个 Controller 暴露接口，Service 层承载业务逻辑，Mapper 层负责与数据库交互，DTO/VO 用来承接请求与返回，Redis 则承担了远超“普通缓存”的职责。项目的整体调用链非常清晰：前端请求先进入拦截器体系，再由 Controller 交给 Service，Service 视业务需要访问 MySQL、Redis、Redisson 或 Lua 脚本，最终统一返回 Result。这个结构之所以重要，是因为它既保留了教学项目的清晰度，又已经具备了工程化系统的基本层次感。

在功能层面，用户端是最完整的一条主线。用户可以按分类、名称、距离等方式浏览店铺，查看店铺详情、营业信息、优惠券、评论和关联博客，并通过搜索和分类页快速定位自己感兴趣的商户。用户看到合适的优惠券后，可以直接发起购买，系统生成订单并进入待支付状态，随后再模拟支付，支付完成后订单进入可核销状态，生成核销凭证，最终由门店店员完成核销。除了交易链路，用户端还支持内容社区能力：用户能够发布博客、给博客点赞、评论博客、关注别的用户、查看关注流、给店铺打分和留言。这里的博客不是孤立的文本内容，而是允许关联到具体商户，构成“内容种草 + 店铺转化”的闭环，这也让项目比单纯的团购系统更丰富。

店员端和店长端是这个项目业务深度比较明显的部分。店员登录后只会看到自己所属门店的数据，包括待核销订单、已核销订单、待支付和已取消订单等视图，并且只能操作与本店相关的记录。核销时，后端不只是做一个简单的状态修改，而是会校验订单当前状态、核销权限、核销码以及门店归属，确保只有处于正确状态的订单才能被正确门店的店员核销。店长端继承店员能力，同时增加了门店资料维护和员工管理能力。这样一来，管理员不需要包揽所有门店内部管理动作，而是通过“管理员建店并指定首个店长，店长再管理本店员工”的链路，让权限边界更加贴近真实业务组织结构。

平台管理端是这个项目从“前台应用”走向“可运营系统”的关键。管理员能够登录专属后台，查看整体统计卡片，新建店铺，并在建店时同步创建首个店长账号。新建店铺并不是简单插一条 `tb_shop` 记录，而是一个多对象联动动作：需要写入店铺表、同步 Redis GEO 坐标、初始化门店与店长关系，并补充布隆过滤器中的店铺 id，避免新建店铺在前台被误判为非法 id。管理员侧不再直接承担日常店员管理，而是主要负责平台级初始化与治理，这让项目的角色划分更加稳定，也让管理端从“万能后台”变成“平台级入口”。

如果从数据库模型看，整个项目可以清晰地划分为五条主线。第一条是用户主线，以 `tb_user` 为核心，配套 `tb_user_info`、`tb_follow`、`tb_sign`，承载登录、扩展资料、关注关系和签到能力。第二条是内容主线，以 `tb_blog` 和 `tb_blog_comments` 为核心，形成用户发内容、他人互动、内容关联店铺的社交闭环。第三条是店铺主线，以 `tb_shop`、`tb_shop_type`、`tb_shop_comment` 组成，用于表达商户本身、商户分类与用户评价。第四条是交易主线，以 `tb_voucher`、`tb_seckill_voucher`、`tb_voucher_order` 组成，贯穿优惠券、秒杀、下单、支付、核销等核心业务。第五条是权限主线，以 `tb_shop_clerk` 和 `tb_admin` 为核心，把普通用户、店员、店长、管理员从数据层面彻底分开。这几条主线交织在一起，形成了“用户发现内容与店铺 -> 用户购买优惠券 -> 门店核销订单 -> 平台与店长完成管理”的整体结构。

技术深度最先体现在登录与权限设计上。项目没有把所有角色混用一套登录态，而是拆成用户、店员、管理员三套登录体系，并分别对应独立的拦截器、Holder 和 Redis 登录态。请求进入系统后，首先由全局刷新拦截器处理 token 续期，再根据路径命中用户端、店员端或管理端登录拦截器，最后由各自的 Holder 把当前身份对象注入到线程上下文中。这样一来，业务方法里可以直接通过 `UserHolder`、`ClerkHolder`、`AdminHolder` 获取当前身份，而不需要把用户 id 或店员 id 从前端层层传递。这种设计同时提升了安全性和代码可读性，也让“当前登录人是谁”这个问题在整个系统里有了统一答案。

第二个技术深度来自 Redis 的多用途使用。Redis 在这个项目里并不是一个简单的热点数据缓存，而是承担了四种完全不同的职责。第一，登录态存储。用户、店员、管理员登录成功后，后端生成随机 token，把轻量 DTO 形式的身份信息写入 Redis Hash，再通过拦截器恢复到 Holder 中。第二，缓存与防穿透。店铺详情查询使用布隆过滤器过滤明显非法的 id，再配合逻辑过期和互斥锁解决热点店铺缓存重建问题；同时项目中还封装了 `CacheClient`，将缓存重建、逻辑过期、空值缓存等策略统一了起来。第三，GEO 查询。店铺根据经纬度写入 Redis GEO，前台即可按距离获取店铺列表，而不必完全依赖数据库复杂地理查询。第四，Feed 流与关注流。用户发博客后，博客 id 会被推送到粉丝收件箱的 `ZSet` 中，粉丝读取关注流时直接从自己的 feed 中按时间倒序拉取博客 id，再回表补齐博客详情。这已经不是“普通缓存”，而是典型的有序时间线索引设计。

交易链路体现了项目最强的一段业务与技术结合能力。普通优惠券与秒杀优惠券共用一套主表结构，秒杀能力通过扩展表与 Lua 脚本实现高并发下的一人一单和库存预扣减。下单成功后生成订单，订单在后续会经历待支付、已支付、已核销、已取消等状态变化。支付、取消与核销并不是简单粗暴地改一个字段，而是依赖状态判断和条件更新确保只有符合当前状态的订单才能被合法流转。最近项目又在这基础上补充了 Redis 层的短期防重复提交，形成了“Redis 控入口，状态条件更新保结果”的幂等组合。也就是说，快速连点不会让重复请求都进入深层业务逻辑，而就算绕过 Redis，数据库层的条件更新仍然能够保证最终只生效一次。这个设计很适合拿来说明项目在并发与一致性方面的考虑不是停留在表面。

为了继续增强工程质量，项目还引入了两项比较有代表性的技术改进。第一是统一参数校验。原先很多接口在 Service 层里手写了大量空值、长度、格式判断，现在逐步将这些“参数本身是否合法”的检查迁移到 DTO 和请求体注解中，由 `@Valid`、字段校验注解和全局异常处理统一兜底，让 Service 更专注于业务规则，比如是否存在、是否有权限、当前状态是否允许流转。第二是操作日志系统。项目增加了 `tb_operation_log` 表和基于 AOP 的操作日志切面，通过自定义注解 `@OperationLogRecord` 记录管理员建店、创建店长、店长创建员工、修改店铺资料、店员核销订单等关键动作。日志不仅包含操作人类型、操作人身份、所属模块和动作，还记录请求方式、路径、来源 IP、执行时间和成功失败状态。更进一步，切面不仅识别抛异常的失败，还能识别 `Result.fail(...)` 这种业务失败，使日志真正反映了系统行为，而不是简单记录“有没有异常”。

从工程实现角度看，这个项目已经不只是一个功能堆砌的课程作业，而是有比较明显的系统化意识。首先，权限边界被认真地区分了出来，用户、店员、店长、管理员不是靠页面区分，而是后端真正隔离。其次，Redis 的使用覆盖登录态、缓存、时间线、GEO 和高并发控制，体现出它被当成基础设施而不是点缀。再次，项目已经开始出现统一校验、统一异常处理、AOP 日志、幂等防重这些偏工程质量的内容。最后，业务上又具备店铺、内容、交易、后台管理三条线并行的复杂度，能支撑比较完整的系统讲述。

如果把整个项目压缩成几条最值得展示的主链，可以概括为四条。第一条是“用户内容链”：用户登录之后，完善资料、发布博客、关联店铺、他人点赞评论、粉丝在关注流中看到内容，这体现了用户、内容与 Redis Feed 的联动。第二条是“用户交易链”：用户浏览店铺、领取或购买优惠券、生成订单、模拟支付、查看核销码，这体现了交易主线的完整性。第三条是“门店履约链”：店员登录后查看本店订单，使用核销码核销已支付订单，这体现了订单状态流转、权限校验和幂等处理。第四条是“平台建店链”：管理员新建店铺并指定首个店长，系统同步写入店铺数据、GEO 信息、门店店长关系和布隆过滤器，之后由店长继续维护本店资料并创建员工，这体现了后台治理与门店自治的结合。

就当前完成度而言，这个项目已经具备较强的展示和讲解能力。它既能从用户视角讲“本地生活 + 内容种草 + 团购消费”，也能从平台视角讲“多角色权限 + 门店后台 + 订单履约”，还能够从技术视角讲“Redis 深度使用 + 缓存与逻辑过期 + 秒杀 Lua + 幂等 + AOP 日志 + 统一校验”。如果后续还想继续拉高深度，可以在不改变大框架的前提下进一步完善统一错误码、事务后事件处理、操作日志查询页、缓存重建排序严谨性，以及更完整的状态机文档。不过即使不继续扩展，这个项目也已经具备一个完整系统该有的大部分骨架和关键亮点。

总的来说，城味项目的价值不在于单个接口多复杂，而在于它把“店铺、内容、优惠券、订单、核销、管理端、店长端、店员端”这些看似分散的元素串成了一套真正闭环、角色清晰、技术栈扎实的业务系统。它既能作为一个前后端联动的完整实践项目，也能作为一个很适合讲权限、缓存、并发、履约和工程化思路的综合案例。
